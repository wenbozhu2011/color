## A general spec for REST-based conversational communication between a client and server

## Use cases

A client represents a user application which interacts with a remote server endpoint. 
The exact server endpoint implementation is not important, e.g. a single stateful process, 
a distributed service with a storage backend for application-level states.

The application-level semantics of the client-server communication is stateful in nature due to 
explicit or implicit causality between client-sent and server-sent messages and their ordering. 
That is, there is no spontaneous client-sent or server-sent messages, e.g. one-way notification or signaling. 
The most obvious example would be a "high-hz chatbot" service. 

## Background: why REST (stateless RPC)

A naiive implementation may map the client-server communication to a single bidi stream, over which all 
client-sent and server-sent messages are strictly ordered. A bidi stream creates a stateful API (for the service), which 
is often less reliable and scalable compared to stateless RPC based APIs.

Moreover, the causality between client-sent and server-sent messages is not understood in any way by the
API spec (metadata) or the underlying serving infra (frameworks, proxies), except when the entire bidi stream
represents a strictly serialized input->output relation, e.g. a video transcoding service.

As a result applications may be forced to annotate messages to express request/response relationship between
messages and hence render the bidi stream largely a tunneling API.  

This problem motivates the creation of this spec, which only use (non-streamed) RPCs to support otherwise stateful 
conversational communication.

## Communication properties (beyond exactly-once)

1. Client sends requests in sequence. The ordering of requests is signficant to both the client and server application.

2. For every request, a response (inc. application-level error) has to be received. Requests may be retried by the client
until a response is received. When a response is received by the client, the same response should have been seen by the server (obviously).

3. When a new request is generated by a client, all the responses (from previous requests) that are known to the client "happens before" 
the generation of this request. The ordering of those responses is also signficant. 

4. Parallel requests are supported, that is, with responses pending. Their responses may come in a different order.

5. Request or responses may be lost arbitarily, independently of each other, due to networking, infra, application reasons. When a request or
response fails to reach the peer, the client will retry the request immediately. The exact failure notification mechanism is implementation specific.

6. The server may delay a request due to message ordering or gaps. Once a response is generated by the server, the response will be 
sent back to the client immediately.

6. The server will buffer a response until the client has received by the response. For any request, a response should only be generated once
if the client has received the response. That is, a request should not be re-processed on the server-side for the purpose of recovering a
lost response.

## Implementation concerns

The client-side implmentation should not require any form of "rich client". That is, a manual REST client should be able to implement the
spec easily. Request retry should be handled by the underlying transport layer, e.g. a HTTP client with built-in retry support.

To support high message rates, reduce user-visible latency, and improve service efficiency, we expect the server to leverage 
in-memory state for supporting such a conversation protocol. This will require all the requets to be sent to a single server process until 
a failover happens due to load distribution, machine draining or process failures. The conversation protocol should enable automatic
server process failover with minimum support required from the client, without breaking any safety properties. 

## Verification with a reference implementation  

A single client generates a squence of requests to a single server process. The request and response may be dropped arbitarily.

To confirm the correctnes of the protocol design and its implementation, the verification should check
- client/server progression, i.e. liveness upon failiures
- request/response histories match between the client and server, i.e. the server knows exactly what the client sees
- retransmission buffer growth is limited  

## Server failover

Server failover may be triggered by the server process itself or the server process may crash or become unreachable. 

To support failover, the server may periodically checkpoint and save the message history, e.g. via distributed storage . 

When a failover happens, the server may fail a request (new or retried) with a special status code (5xx) for the client to 
re-send any previous requests and responses known to the client (but not to the server). Client-retried requests 
will be reprocessed by the new server.

## HTTP support

The conversation protocol guarantees exactly-once semantics. As a result, the HTTP client (as a REST implementation) needs to
allow POST to be retransmitted upon any request or response delivery failures. 

For simplicity, batching is not supported. However, the failover recovery protocol requires the client to encode multiple requests and responses 
as message history. A dedicated request may be used which will encode request/response history in a single HTTP request body
e.g. with a JSON envelope.

There is no special requirement on how to route HTTP requests to the same server process. When requests start to be routed to a different process
(consistenly), the failover process will be triggered.
